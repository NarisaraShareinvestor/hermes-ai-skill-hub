🎯 B · Skill-Driven Knowledge ใหม่

ทุก skill เป็นตัวขับ (รวม skill ส่วนตัว) — แต่ละตัวบอกเองว่าต้องเรียนอะไร ไม่เรียนมั่ว

  • publish → คลังองค์กร · ส่วนตัว → คลังเจ้าของ
  • คัดกรอง: เกี่ยวกับ skill นี้จริงไหม
  • เก็บผูก skill_id → ใช้ตอนรัน skill

4 มิติที่เรียน (ตรงกับที่เลือกไว้)

มิติเรียนจากอะไรเก็บที่ไหนนำไปใช้ยังไง
🎨 สไตล์/ฟอร์แมต ความยาวคำตอบที่ผู้ใช้ตอบรับดี, ขอ bullet/ตาราง/ย่อหน้า, EN vs TH, โทนทางการ/กันเอง PREFERENCE prompt: "ผู้ใช้ชอบคำตอบ สั้น เป็น bullet ภาษาไทย"
🔤 ศัพท์/บริบทงาน ชื่อบริษัท/โปรเจค/ทีมที่พูดถึงบ่อย, ตัวย่อเฉพาะ, แผนกของผู้ใช้ FACT prompt: รายการศัพท์ + ความหมาย → ตอบโดยไม่ต้องถามซ้ำ
🔁 งานที่ทำซ้ำ intent ที่เกิดซ้ำ (เช่น "สรุปประชุมเป็น MOM", "ร่างอีเมลแจ้งมติ") BEHAVIOR เสนอ shortcut/skill ส่วนตัว (ต่อจาก auto-skill v6 เดิม)
👍 คุณภาพจาก feedback FeedbackEvent 👍/👎 + comment ของ user คนนั้น USER_PLAYBOOKใหม่ "do/don't" รายคน เช่น "อย่าใส่ disclaimer ยาว — เคยโดน 👎"
USER_PLAYBOOK = MemoryType ใหม่ 1 ตัว — เก็บ "กติกาการตอบเฉพาะคน" ที่กลั่นแล้ว (JSON: {style, glossary, recurring, dos, donts, updated}) เพิ่มผ่าน _run_migrations (ALTER TYPE) เหมือนตอนเพิ่ม FACT/BEHAVIOR — ความเสี่ยงต่ำ มี pattern แล้ว
ตัวอย่างผล — Hermes ของ "คุณนริศรา" หลังเรียน 1 สัปดาห์
สรุปประชุมเมื่อกี้ให้หน่อย
[จำได้ว่าชอบ MOM สั้น เป็น bullet + ใส่ action item พร้อมผู้รับผิดชอบ ไม่ต้องมีคำนำ]
• มติ: อนุมัติงบ Q3 …
• Action: คุณสมชาย ส่งร่างศุกร์นี้ …
ไม่ต้องสั่งฟอร์แมตซ้ำทุกครั้ง — เพราะ playbook รายคนถูกฉีดเข้า prompt

3 ชั้น B — ดูดความรู้ภายนอก ตามทุก Skill

หลักสำคัญ: ไม่เรียนมั่ว — ทุก skill (publish + ส่วนตัว) เป็นตัว "สั่งงาน" ว่าจะไปเรียนเรื่องอะไร ความรู้ผูกกับ skill ตัวนั้นโดยตรง เพื่อให้ skill นั้นเก่งขึ้น — ใครสร้าง skill ส่วนตัว ระบบก็เรียนให้ด้วย

🎯 Skill-Driven Knowledge Ingestion

แต่ละ skill มี tags + description + department อยู่แล้ว → ระบบแปลงเป็น "หัวข้อที่ต้องไปเรียน" → ดึงมาเฉพาะเรื่องนั้น → ผูกกลับเป็นความรู้ของ skill (scope ตาม visibility ของ skill)

published + private + team → tags/description = ขอบเขต
🔎

WebSearch ด้วยคำค้นที่กลั่นจาก skill นั้น
🧹
Curate

🔗

skill_id + visibility (private = เห็นเฉพาะเจ้าของ)
กลไกกัน "เรียนมั่ว": ก่อนเก็บทุก doc ต้องผ่านด่านคัดกรองด้วย LLM ว่า "เนื้อหานี้ทำให้ skill ‹X› ตอบได้ดีขึ้นจริงไหม" ถ้าไม่เกี่ยว = ทิ้ง. คำค้นทุกคำมาจาก tags/description ของ skill ไม่ใช่หัวข้อสุ่ม
skill ส่วนตัว: ความรู้ที่เรียนมา visibility=private ผูกกับ owner — เห็นเฉพาะเจ้าของ ไม่รั่วให้คนอื่น ถ้าเจ้าของ publish ภายหลัง คลังของ skill จะถูกยกเป็นระดับองค์กรตาม visibility ใหม่ พร้อม skill

ตัวอย่าง: skill ใน Store → เรียนอะไร (อิง _CATALOG_SKILLS จริง)

4 สิ่งที่ต้องเพิ่มในระบบ (data model)

น้อยที่สุดเท่าที่จำเป็น — เสริมของเดิม ไม่รื้อ

รายการชนิดหน้าที่
MemoryType.USER_PLAYBOOKใหม่ (enum + migration)กติกาการตอบเฉพาะคนที่กลั่นแล้ว (ชั้น A)
KnowledgeDocใหม่ (ตารางใน observability)1 ก้อนความรู้ — ผูก skill_id + owner + visibility (private/company) + url, summary, fetched_date, status
Skill (เพิ่ม field)มีแล้ว + learning_topicsหัวข้อที่ skill นี้ต้องไปเรียน (กลั่นจาก tags/description ครั้งแรก แอดมินปรับได้)
run_user_learning()ใหม่ (ฟังก์ชัน)ลูปชั้น A รายคน — เรียกใน nightly
run_knowledge_ingestion()ใหม่ (ฟังก์ชัน)ลูปชั้น B — cron เช้า + รวมผลใน report
ImprovementReportมีแล้ว (เพิ่ม field)เพิ่ม user_learning_summary + knowledge_summary ลง JSON
APScheduler jobsมีแล้ว (เพิ่ม job)เพิ่ม cron "ingestion เช้า" + เรียก user-learning ใน nightly เดิม
AI Engineer dashboardมีแล้ว (เพิ่มแท็บ)แท็บ "คลังความรู้" + "การเรียนรู้รายคน" (อนุมัติ/ปฏิเสธ doc)

6 ความปลอดภัย & ธรรมาภิบาล

🛡️ Human-in-the-loop

ความรู้ภายนอกเข้ามาเป็น status=draft — ตอบผู้ใช้เฉพาะ doc ที่ approved (ตามแพทเทิร์น LearnedGuidance เดิม)

🔒 แยก visibility

ความรู้ส่วนตัว (transcript รายคน) ห้ามรั่วเข้าคลังองค์กร — playbook รายคนเห็นเฉพาะเจ้าของ

📎 อ้างอิงเสมอ

ทุกคำตอบจากคลังต้องแนบ source + วันที่ ถ้าไม่มีแหล่ง = ไม่ตอบว่าเป็นข้อเท็จจริงสด

💸 คุมงบ + เรียงลำดับ

เรียน skill ที่ถูกใช้จริงก่อน (skill ส่วนตัวที่ไม่มีใครแตะ → เรียนถี่น้อยลง), budget/วัน, เกินงบ → ข้ามรอบ + alert

⚖️ ToS / ลิขสิทธิ์

เก็บ "สรุป + ลิงก์" ไม่ก็อปเนื้อหาเต็ม, เคารพ robots.txt, เลี่ยง scraping ที่ผิดเงื่อนไข

🗑️ Retention

doc เก่า/ถูกแทนที่ → archive; user playbook แก้/ลบได้ผ่าน memory API เดิม
